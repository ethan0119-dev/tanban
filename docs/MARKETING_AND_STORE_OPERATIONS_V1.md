# 营销应用、快餐码牌与门店营业 V1

本文记录摊伴在对标 FirePOS“应用、店内、设置”三个经营域后形成的一期实现边界。目标不是把当前记录的 19 张应用卡片全部做成空页面，而是先交付优惠券、弹窗广告、抽奖、堂食工作台、快餐码牌和营业时间六条能够持久化、启停、审计与扩展的链路。

## 1. 应用中心与分期

商户后台“营销应用”展示对标系统当前账号可见的营销玩法全集：优惠券、满额立减、配送费满减、下单返红包、弹窗优惠券、弹窗广告、支付有礼、店铺满赠、超级换购、新商品活动、兑换码、券包、收银台、餐桌预定、排队取号、酒水寄存、集点返红包、抽奖活动、门店新客立减。

一期开放：

- 优惠券：代金券和满减券；
- 弹窗广告：首页广告位、排期、频次、动作和事件统计；
- 抽奖活动：免费抽奖，奖品为优惠券或“谢谢参与”；
- 应用中心：按模块数据呈现未配置、草稿、进行中和暂停状态。

配送费玩法、外卖活动范围和外卖订单场景只保留占位或枚举，激活时服务端返回 `DELIVERY_NOT_AVAILABLE`。其他营销卡片显示“规划中”，不伪装成已实现能力。

## 2. 优惠券

### 2.1 活动规则

| 字段 | 约束 |
| --- | --- |
| 类型 | `CASH` 代金券；`FULL_REDUCTION` 满减券 |
| 金额 | 全部使用整数分；优惠额必须大于 0 |
| 门槛 | 满减券门槛必须大于等于优惠额；代金券可无门槛 |
| 发放 | 公开领取；商户发放模式仅保留配置；抽奖专用 |
| 库存 | 总量、已发量和每主体限领 |
| 有效期 | 固定起止时间，或领取后 N 天 |
| 场景 | 堂食、自取；外卖仅预留 |
| 状态 | 草稿、进行中、暂停、结束 |

公开领取和抽奖请求使用 `Idempotency-Key`，重复请求返回同一业务结果。当前小程序 AppID/AppSecret 尚未配置，设备游客键只能作为相关性标识，不能证明微信顾客身份。因此一期领取/中奖资产标记为 `PROVISIONAL`，不会直接参与订单核销。配置 `wx.login` 服务端会话、统一 Quote 与订单优惠快照后，才开放正式券包和抵扣。

需要特别注意：`PROVISIONAL` 领取和中奖仍会占用活动/奖项库存。当前单节点 IP 限流只是防滥用缓冲，不是顾客身份边界；服务端只接受本机 Nginx 覆写的 `X-Real-IP`，直连请求不能用转发头绕过限流，但在微信顾客会话完成前，生产环境仍不应启用有经济价值的公开领券或优惠券抽奖活动。

## 3. 弹窗广告

一期开放五个受控弹窗位置：门店首页 `HOME_POPUP`、商品点单页 `MENU_POPUP`、结算页 `CHECKOUT_POPUP`、订单结果页 `ORDER_RESULT_POPUP`、会员中心 `PROFILE_POPUP`。

- 图片、标题、副标题；
- 起止时间、优先级和状态；
- 每次访问、每日一次、每活动一次；
- 无动作、去点单、去领券、直接领券、打开抽奖；
- 曝光、点击、关闭事件。

同一时间只返回一个优先级最高且动作目标仍可用的活动；营销弹窗出现时不再叠加装修启动页。事件写入失败不阻断首页和点单。

匿名曝光、点击和关闭事件按真实客户端 IP 设置全局与广告位双层限流；同一匿名主体、广告位和事件类型在一分钟内只记一条。该机制用于约束写库体量与基础指标污染，后续多实例部署时由同一缓存接口迁移到 Redis。

## 4. 抽奖活动

- 仅免费参与，不支持现金、余额、付费抽奖或实物自动履约；
- 奖品为 `COUPON` 或 `THANKS`；
- 使用整数权重和密码学安全随机源；
- 每匿名主体的活动日限次按门店 IANA 时区判定，活动期总限次由服务端判定，活动累计 `draw_count` 在同一事务中递增；
- 库存扣减、抽奖记录和优惠券记录在同一事务；
- 幂等键相同的重试返回原结果；
- 已启用活动不直接改奖项；尚未开奖时可暂停后修改，已有抽奖记录时需暂停并复制新活动。

活动页面展示参与规则和次数。正式顾客身份未接入前，中奖优惠券同样是不可核销的临时资产。

## 5. 堂食与快餐工作台

店内订单仍使用同一订单状态机，但运营视图按场景分开：

- 堂食：大桌台卡片突出区域、桌名、订单状态、商品和等待时间；
- 快餐：大取餐号卡片同时显示快餐码牌位置；
- 外卖：保持独立菜单和打印模板，当前禁止创建。

快餐取餐号不再由订单主键取模。系统按 `store_id + business_date` 在数据库事务中分配至少四位、单日持续递增的序号，订单保存取餐号快照。码牌是商户可复用的实体，包含不可猜测的公开 ID、展示编码、名称、状态和二维码 scene；订单保存码牌快照，后续重命名不会改变历史记录。

小程序扫码参数：

```text
普通门店：s=<storeCode>
堂食桌码：tc=<tablePublicScene>
快餐码牌：fp=<fastFoodPlatePublicId>
```

创建快餐订单时服务端重新确认码牌启用、门店归属和营业状态，不能信任本地缓存。

## 6. 营业时间

门店保存 IANA 时区、每周营业计划和临时覆盖状态。每周一天可配置多个时段，支持 `18:00-02:00` 跨夜营业。

营业判定遵守：

- 开始时刻包含、结束时刻不包含；
- 次日凌晨落入前一日跨夜时段；
- 临时营业/临时闭店优先于周计划；
- 门店停用永远不能接单；
- 公共门店接口和创建订单调用同一个判定器。

公共接口返回 `businessStatus`、`businessStatusMessage`、`acceptingOrders` 和 `nextOpenAt`。小程序首页、点单页和结算页显示休息提示并禁用新下单；服务端仍作为最终防线返回 `STORE_CLOSED`。

闭店只阻止创建新订单，不会把营业时已经创建且尚未关闭的订单变成不可支付。小程序和支付接口允许此类订单继续走原支付状态机；订单主动关闭后才拒绝付款。

## 7. 数据与回滚边界

- migration 以向前兼容的加表/加列为主，除旧营业时间迁移外不改写旧经营事实，不重算历史订单；
- 升级时仅把可解析的旧 `HH:mm-HH:mm` 营业时间迁移为周计划；旧值为空或不可解析时保持一周七天 `00:00-00:00` 的全天营业默认，以避免在首次启用强制校验后意外拒单；发布后需由商户确认真实周计划；
- 活动、领取、抽奖、码牌、取餐号和营业时间均携带租户/门店边界；
- 删除/停用活动不会删除历史领取与抽奖记录；
- 回滚应用代码时保留新表和已产生的经营事实，避免破坏对账与审计；
- 外卖能力开放前，激活任何 `DELIVERY` 营销活动或创建 `DELIVERY` 订单请求都必须失败关闭。

## 8. 正式商用前置

1. 配置微信小程序 AppID/AppSecret，完成 `wx.login -> code2Session -> 顾客 Token`；
2. 增加统一 Quote、优惠预占、支付成功核销、关单释放和退款补偿；
3. 在真实 MySQL 5.7 上做并发券库存、抽奖库存和取餐序列集成测试；
4. 对营销图片启用对象存储、域名白名单和内容安全审核；
5. 公开抽奖规则、概率、期限和投诉入口，并经过法务/合规复核。
6. 真实支付启用前，把异常重复扣款的退款对象从“订单”细化到具体支付尝试，并让支付列表按支付尝试展示各自退款额；当前 mock 联调环境不得切换为真实收款。
