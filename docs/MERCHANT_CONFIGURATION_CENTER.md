# 商户配置中心与桌台状态设计

> 版本：2026-07-20。来源为 FirePOS / 爨火当前账号的既有逐页遍历清单、公开前端能力和用户提供的“堂食订单桌台状态”截图。菜单和业务规则只作产品观摩，不复制其商标、文案素材或代码。

## 1. 信息架构

商户后台把配置类能力统一收敛到一级菜单“设置”，子菜单为：

1. 门店设置：门店资料、营业时间、临时营业、紧急闭店。
2. 点餐设置：结算模式、点餐模式、距离校验、顾客字段、备注、支付时效、接单和提醒。
3. 支付设置：支付机构绑定状态、脱敏商户号、费率、结算周期、半屏收银方式和资金安全边界。
4. 通知设置：微信服务号接入状态、商户接收人和通知事件偏好。
5. 隐私与客服：隐私政策、用户协议、客服电话、微信号和客服二维码。
6. 打印设置：门店自动打印总开关和新模板默认触发点。
7. 打印机管理：设备、SN、状态、测试打印和虚拟打印机。
8. 店内打印模板：商家联、顾客联、厨房联、一菜一单、标签等场景。
9. 外卖打印模板：在外卖能力开放前保留独立预配置。
10. 员工与角色：员工账号和 RBAC。

支付与退款流水仍属于经营/财务数据，不并入“支付设置”。桌码与快餐码牌是经营资源，也继续放在“店内”。

## 2. 对标设置项与摊伴处理

| 设置能力 | 摊伴处理 | 原因 |
| --- | --- | --- |
| 未支付订单关单 / 超时继续支付 | 已生效 | 复用 `payment_timeout_minutes` 和 `allow_late_payment`；主动关单后始终禁止支付 |
| 已取消订单迟到支付 | 系统强制隔离 | 不给商户关闭；回调必须查单、验签、核验金额和本地订单状态，异常实收进入人工确认退款 |
| 重复支付 | 系统强制识别与隔离 | 支付尝试追加保存，支付机构流水唯一，保留两笔实收事实并进入支付异常，由有权限人员退款 |
| 库存扣减 | 系统强制 | 下单预占，支付确认，关单/过期释放 |
| 先结账后用餐 | 已生效 | 当前订单和支付状态机的真实模式 |
| 先用餐后结账 | 明确预留 | 需要开台、并台、餐中加菜、结账、逃单风控，不提供无效开关 |
| 多人点餐 | 已生效 | 同一桌允许不同顾客分别下单 |
| 单人点餐 | 已生效 | 同桌存在其他顾客活动订单时服务端拒绝新顾客提交 |
| 判定用户距离 | 已生效 | 小程序请求 GCJ-02 定位，服务端按门店经纬度和最大距离重新计算；缺少定位时失败关闭 |
| 必填手机号、整单/单品备注 | 已生效 | 小程序按门店设置展示字段，服务端再次校验 |
| 催单、自提核销、评价 | 配置位/能力预留 | 还没有完整消息、核销和评价审核链路，页面明确标记预留 |
| 紧急闭店 | 已生效 | 通过有截止时间和原因的营业状态覆盖实现，不用永久布尔值污染周计划 |
| 隐私政策、用户协议、私人客服 | 已生效 | 商户端维护；小程序“我的 → 隐私与协议”和“联系商家”读取公开配置 |
| 微信服务号通知 | 偏好已落库，发送未开放 | 还需要平台服务号 AppID/Secret、消息模板、商户接收人关注和 OpenID 绑定 |
| 短信、云喇叭、语音合成 | 能力清单保留 | 属于收费/硬件通道，不能用浏览器在线提示冒充可靠通知 |
| 标记支付 | 暂不开放 | 它是线下记账标记而非支付机构到账证明，需要独立权限和对账审计 |

## 3. 支付设置逻辑

商户支付页只读展示：

- 支付机构：`mock`（开发）或 `tianque`（会生活 / 随行付）。
- 绑定状态：开发环境、待平台进件、已绑定。
- 脱敏商户号和小程序子 AppID 配置状态。
- 签约费率与结算周期；码农咖啡基线为 `0.38% / T+1`，按租户保存，后续必须以实际协议为准。
- 收银方式：会生活半屏收银台。
- 资金流向：收单机构直接结算到商户银行卡，摊伴不过款。
- 支付确认：支付机构回调验签 + 主动查单/对账补偿。
- 支持部分退款。

商户端不能修改平台机构号、私钥、公钥、通知密钥、银行卡或完整商户号。上述参数由平台管理端或服务器密钥配置维护。

## 4. 微信服务号通知边界

服务器预留环境变量：

```text
TB_WECHAT_OFFICIAL_ACCOUNT_APP_ID
TB_WECHAT_OFFICIAL_ACCOUNT_APP_SECRET
```

两者必须成对配置。当前商户端可选择 `ORDER_PAID`、`REFUND_CREATED`、`PRINT_FAILED`、`STORE_EXCEPTION` 事件并保存接收人备注，但接口始终返回明确的三段状态：平台参数、商户接收人绑定、消息投递。只有三段都完成后才能标记投递启用。

后续实现顺序：服务号凭证托管 → 模板配置 → 商户扫码关注/绑定 OpenID → Outbox 消费者 → 微信发送结果和重试/死信 → 商户可见的发送记录。

## 5. 堂食桌台状态

堂食订单新增“桌台状态”页签，按桌台区域显示大卡片：

- `UNOPENED / 未开台`：没有活动堂食订单。
- `OPENED / 已开台`：存在待付款或已付款但尚未进入制作的订单。
- `DINING / 就餐中`：订单处于已接单、制作中或待出餐。

订单完成、关闭或退款清台后不再占用桌台。看板底部显示当前结算模式和点餐模式。点击有活动订单的桌卡可直接查看订单详情。

当前状态来自订单事实，不用浏览器内存猜测。若后续要精确覆盖“顾客只扫码选桌但尚未下单”的已开台状态，需要新增 `dining_session`、扫码确认、心跳和过期清理，本期不把一次匿名扫码直接记作已开台。

## 6. API 与数据

新增商户 API：

- `GET/PUT /api/v1/merchant/operation-settings`
- `GET /api/v1/merchant/payment-settings`
- `GET /api/v1/merchant/table-board`

新增 `store_operation_settings`，按 `tenant_id + store_id` 隔离；客服二维码必须来自本租户图片库。`tenants` 增加 `payment_fee_bps` 和 `payment_settlement_cycle`。公开门店接口只返回顾客需要的点餐约束、协议和客服联系信息，不返回支付密钥或服务号凭证。

## 7. 安全约束

- 距离开关启用但未配置合法门店坐标时拒绝保存。
- 下单时启用距离校验但顾客未授权定位，服务端拒绝订单。
- 商户前端提交的门店、坐标、桌码、金额和支付结果都不能作为最终信任来源。
- 支付页不得显示完整商户号、银行卡或密钥。
- `PAY_AFTER_MEAL` 等未完成能力必须禁用并标注预留，不能保存后仍按原逻辑执行。
- 所有设置写入均经过租户边界、经理角色权限和审计日志。

## 8. 商品属性库

商品配置中心提供门店级“属性库”，统一维护温度、甜度、辣度等不决定库存的点单选项。属性定义包含单选/多选规则、最少和最多选择数、属性值、加价、默认值、排序及启停状态。

商品既可以引用属性库，也可以保留商品专属属性。引用属性库时，`product_option_groups.attribute_group_id` 和 `product_option_values.attribute_value_id` 保存来源关系，同时继续物化商品级选项 ID，保证现有小程序选项提交、服务端金额复算和订单配置快照兼容。属性库更新会在同一事务中同步所有已关联商品；仍被商品引用的属性定义禁止删除。
