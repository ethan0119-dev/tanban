# 首页图片热区与结构化打印模板

本文记录 2026-07-20 对标 FirePOS 装修热区、商家联/顾客联/厨房联/标签模板后，摊伴系统采用的数据结构、交互边界和设备适配方式。

## 1. 单图热区模块

首页新增 `HOTSPOT_IMAGE` 模块。商户上传或从素材库选择一张完整设计图，再在图片上拖拽矩形热区；小程序只叠加透明点击区域，不拆解或重画图片中的视觉内容。

```json
{
  "type": "HOTSPOT_IMAGE",
  "config": {
    "imageUrl": "https://tbapi.example/api/v1/public/media/uploads/t1/s1/2026/07/random.png",
    "alt": "码农咖啡首页导航",
    "hotspots": [
      {
        "id": "hotspot_uuid",
        "x": 4.2,
        "y": 36.5,
        "width": 28.4,
        "height": 18.8,
        "label": "堂食点单",
        "action": { "type": "OPEN_MENU" }
      }
    ]
  }
}
```

`x/y/width/height` 均为相对图片的百分比，因此同一份配置可以适配不同手机宽度。服务端限制坐标范围、热区数量、最小尺寸和越界；前端绘制结果只能使用动作白名单：

- `NONE`：无动作；
- `OPEN_MENU`：进入点单菜单，并保留当前桌码上下文；
- `OPEN_ORDERS`：进入订单列表；
- `OPEN_PROFILE`：进入个人中心；
- `CALL_PHONE`：调用微信拨号能力，号码仍需通过格式校验。

不接受任意 JavaScript、任意小程序路径或外部 URL，避免商户装修配置演变为脚本注入和开放跳转入口。

### 1.1 图片上传

商户后台通过认证接口上传图片，服务端校验文件头、MIME、扩展名和体积，生成不可预测文件名并写入持久目录。上传目录通过环境变量配置并挂载为持久卷；公开读取只暴露生成后的相对路径，不允许客户端决定磁盘路径。

素材记录仍归属当前租户和门店。删除素材记录不会自动破坏已经发布的装修版本；物理文件清理应在确认没有草稿、已发布版本或历史版本引用后异步执行。

## 2. 打印模板的三个层次

打印能力拆成三个层次：

1. 商户编辑的是厂商无关的结构化模板；
2. 服务端根据订单快照渲染为 58mm 或 80mm 定宽票据，并固化到打印任务；
3. 打印机适配器把任务转换为芯烨云打印 API、ESC/POS 或其他厂商指令。

这样页面预览、虚拟打印机和未来真实硬件共享同一份模板语义，商户不需要在更换打印机后重新装修票据。

## 3. 模板作用域与联次

模板唯一作用域为：

`tenant_id + store_id + business_type + template_type + copy_role`

一期支持：

| `template_type` | `copy_role` | 用途 | 默认状态 |
| --- | --- | --- | --- |
| `RECEIPT` | `MERCHANT` | 商家留存整单、金额、支付和桌台信息 | 启用 |
| `RECEIPT` | `CUSTOMER` | 顾客联，可显示顾客、地址、金额和二维码 | 关闭 |
| `RECEIPT` | `KITCHEN` | 厨房整单，突出规格、加料和备注，默认隐藏价格 | 关闭 |
| `LABEL` | `ITEM` | 按订单明细数量拆分杯贴或餐品标签 | 启用 |

旧的 `RECEIPT` 自动迁移为 `MERCHANT`，旧的 `LABEL` 自动迁移为 `ITEM`。顾客联和厨房联默认关闭，避免升级后无意中多打纸张。

对标系统还存在一菜一单、储值、第三方核销和收银台条形码等模板。这些能力保留在产品能力清单中，但不混入当前订单打印模型；启用相应业务域时再增加明确的 `template_type/copy_role`，不能复用厨房联或标签联冒充。

## 4. 结构化布局

模板保存纸宽、份数、触发点和 `layout`。布局当前包含：

- 抬头样式、字号、店铺名称、订单类型、订单号、取单号和桌台；
- 商品明细、规格加料、价格、实付和支付方式；
- 订单备注、顾客信息、配送地址和取单二维码；
- 受长度限制的自定义抬头与底部文案。

服务端以订单与商品快照为数据源，不能回查已经变化的商品名、规格或桌台名称来重构历史小票。每个打印任务同时固化 `template_id`、`copy_role`、`paper_width` 和最终内容，便于补打、排障和审计。

纯文本 `content` 继续保留为兼容回退：旧模板的 `layout={}` 时按原变量模板渲染；新模板优先使用结构化布局。未知变量不获得执行能力。

## 5. 任务生成与补偿

- 订单创建、支付成功和退款成功的业务事务只落订单/资金事实，并在同一事务写入带唯一去重键的 `print_outbox`；模板查询、排版和打印任务生成由后台 worker 异步完成，打印故障不会回滚已经确认的支付或退款状态；
- outbox 事件与生成的 `print_jobs` 在同一事务内完成，失败时回滚本次生成的全部任务并指数退避重试；超过上限进入 `DEAD` 并记录错误，便于告警和人工处理；
- 自动打印先按门店的小票/标签总开关过滤，再读取与订单 `order_type` 完全一致的已启用模板；
- 每台打印机显式配置负责的联次。小票机可选择商家联、顾客联、厨房联，标签机只负责商品标签；任务生成时按该配置过滤，避免前台与后厨打印机互相串联或重复出纸；
- 模板纸宽用于编辑和预览，最终任务始终按物理打印机的 58mm/80mm 宽度重新排版并固化，避免把 80mm 内容直接发送给 58mm 设备造成截断；
- 同一种输出设备可以按已启用联次生成多条独立任务，某一联失败不覆盖其他联次结果；
- 标签继续按“订单明细 × 商品数量 × 模板份数”拆分；
- 商家联、顾客联和厨房联按各自模板份数生成整单任务；
- 打印任务保留失败、未知和手工补打状态；超时属于结果未知，不能自动无限重试而造成重复出纸；
- 补打票据必须带有“补打”标识并写入审计日志。

当前虚拟打印机用于验证任务、排版和重试流程。接入芯烨 T58H 前，还需要用真实厂商文档确认二维码、加粗/倍高、切纸、语音提醒、任务查询和幂等字段，再由适配器映射结构化模板，不修改订单域。

outbox 的去重粒度分别使用支付记录 ID、退款记录 ID 和订单 ID。它防止支付机构重复回调或 worker 重试时重复生成同一业务事实对应的打印任务；商户主动点击“补打”仍是另一条显式操作，任务必须带补打标记和操作人审计。
