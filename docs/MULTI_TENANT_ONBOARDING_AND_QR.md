# 商户开通、首账号与共用小程序二维码

## 1. 商户开通是一个原子流程

平台新增商户时必须同时创建三类数据：

1. `tenant`：SaaS 商户租户；
2. `store`：该商户的首门店，包含全局唯一 `store.code`；
3. `user`：首个 `MERCHANT_OWNER` 老板账号。

三者在同一个数据库事务中提交，任一步失败则全部回滚。平台创建完成后只在结果弹窗中展示一次初始密码，并同时交付运营后台地址、登录账号、首门店名称与门店码。密码只保存 bcrypt 哈希，关闭结果弹窗后不能反查原密码。

历史数据或异常导入造成商户没有老板账号时，平台商户详情提供“创建首个老板账号”入口。接口只允许在该租户不存在 `MERCHANT_OWNER` 时执行，避免无意创建多个最高权限账号。

商户状态支持：

- `ACTIVE`：可登录运营后台并对外营业；
- `PENDING`：资料待完善，不允许作为正式营业商户使用；
- `DISABLED`：停用。

## 2. 多商户共用同一个微信小程序

所有商户共用同一 AppID。二维码不切换小程序，而是通过启动参数选择门店：

```text
pages/home/index?scene=s%3D<storeCode>
```

小程序解析 `scene` 中的 `s=<storeCode>`，随后所有公开请求都使用：

```text
/public/stores/{storeCode}/...
```

服务端通过全局唯一 `store.code` 找到 `store_id` 和 `tenant_id`，再按这两个作用域读取装修、商品、活动与订单。购物车、优惠券和本地订单索引也按 `storeCode` 分区，避免切换门店后串数据。

普通门店码只负责选择公开门店，因此 `storeCode` 不是权限凭证。涉及桌台和快餐码牌的入口使用服务端生成的不可猜随机令牌：

```text
tc=<tablePublicScene>
fp=<fastFoodPlatePublicId>
```

小程序先把随机令牌交给服务端解析，服务端返回权威门店与桌台/码牌上下文；创建订单时再次校验其租户、门店和启用状态。客户端传来的 `tenant_id/store_id/table_id` 不能作为可信归属依据。

## 3. 二维码类型

| 场景 | scene | 用途 |
|---|---|---|
| 门店首页 | `s=<storeCode>` | 进入指定商户门店，普通自取点单 |
| 堂食桌码 | `tc=<opaqueToken>` | 解析门店和桌台，订单固化桌台快照 |
| 快餐码牌 | `fp=<opaqueToken>` | 解析门店和取餐放置位置 |

商户后台当前展示的是用于联调的普通二维码。正式物料必须由平台服务端调用微信 `getUnlimited`，使用同一 AppID、不同 scene 生成官方小程序码。AppSecret 只保存在服务端，不下发到浏览器或小程序。

## 4. 冷启动与防串店

- 扫描新门店码时清理旧桌码、快餐码牌和门店主题缓存；
- 从“最近使用的小程序”冷启动且没有 scene 时，回到配置的默认测试门店；
- 无效、停用或跨门店的桌码/码牌必须阻断下单，不能静默降级到默认门店；
- 支付、退款和打印以订单中已经固化的 `tenant_id/store_id` 为准，不以后续前端参数覆盖。
