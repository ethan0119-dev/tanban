# 商户运营后台 V2：商品、用户会员与店铺装修

本文记录 2026-07-20 在一期正式代码上新增的三个经营域。对标功能名称来源于 FirePOS 商户后台的实时菜单清单；摊伴实现以可验证的业务闭环为准，不把仅有菜单或静态表格的占位页标记为完成。

## 1. 商品管理

### 1.1 菜单与数据含义

| 对标能力 | 摊伴 V2 实现 | 点单端影响 |
| --- | --- | --- |
| 商品管理 | 延续商品、SKU、价格、库存、上下架 | 商品与可售 SKU 公开展示 |
| 分类管理 | 分类新增、编辑、停用、排序 | 决定小程序分类导航与商品归属 |
| 套餐商品 | 固定套餐资料、售价和组成 JSON 维护 | 本版不做分组任选与组件库存联动 |
| 加料商品 | 加料名称、图片、默认加价、状态 | 作为顾客可选加料参与服务端计价 |
| 加料组 | 最少/最多选择数、包含的加料 | 小程序按组展示并校验选择约束 |
| 规格 | SKU 继续作为决定价格和库存的规格 | 顾客选规格后定位唯一 SKU |
| 属性 | 商品私有属性组和值，可单选/多选、必选/可选、加价 | 甜度、温度、冰量等进入统一配置弹层 |
| 单位、标签、打印标签、备注 | 通用商品资源库与商品绑定 | V1 保存基础数据和绑定；筛选、快捷备注与真实打印路由后续启用 |
| 规格/属性/加料模板 | 版本无关的配置模板资料 | V1 维护模板资料，自动套用/复制能力后续启用 |
| 临时商品库 | 临时商品资料维护 | 本版不直接生成正式商品 |

### 1.2 权威计价

顾客端只提交 `skuId`、属性值 ID、加料组/加料 ID 和数量，不提交可信成交价。创建订单的数据库事务会重新验证：

1. SKU、商品、租户和门店归属；
2. 属性是否属于商品，是否满足 `minSelect/maxSelect`；
3. 加料组是否已绑定商品，加料是否属于该组；
4. SKU 库存是否足够；
5. `SKU 基础价 + 属性加价 + 加料加价`。

最终把选项名称、当时价格和单品备注写入 `order_items.configuration_json`。商户后续改名、改价或停用加料不会改变历史订单或补打内容。购物车行键同时包含 SKU、属性、加料和备注，相同商品的不同口味不会错误合并。

### 1.3 暂缓能力

- 分组任选套餐与套餐组件库存联动；
- 加料独立库存；
- 渠道差异价、会员价和优惠叠加；
- Excel 导入与模板变更自动传播；
- 可视化打印标签画布。
- 每日营业自动补库存（日切幂等任务）；当前运营端明确显示为“预留”且禁止开启，库存仍由人工维护。

## 2. 用户、会员与储值

顾客实体使用 `customers`，与后台登录账号 `users` 严格分离。一级菜单“用户管理”下的页面覆盖：

- 用户列表与详情；
- 标签管理与顾客标签分配；
- 会员卡等级；
- 开卡记录；
- 会员卡设置；
- 等级订单；
- 余额流水；
- 储值规则；
- 储值记录；
- 储值设置。

微信 `openid/unionid`、游客键和系统顾客号均按大小写精确匹配并使用二进制排序规则；它们是不可改写的身份标识，不能用数据库默认的不区分大小写比较合并顾客。

### 2.1 资金约束

- 金额统一使用整数分；
- `balance_ledger` 是资金事实源，`balance_accounts` 只是同事务更新的余额投影；
- 流水没有编辑和删除接口，纠错只能新增调整/冲正记录；
- 余额账户使用 `SELECT ... FOR UPDATE` 串行变动，禁止变成负数；
- 每次变动必须携带 `Idempotency-Key`，重复请求返回同一结果；
- 手工调账和手工登记储值仅老板角色可执行，店长只管理资料和查看流水；
- 操作原因、前后余额、操作者和幂等号进入同一事务及审计记录。

当前储值记录是商户侧的人工登记/台账能力，不伪装为小程序真实充值。正式开放顾客储值前，必须完成顾客微信会话、通道支付验签回调、退款冲正和风险审核。

订单支付尝试采用追加记录：同一订单超时关闭后重新拉起支付，会生成新的支付尝试号和通道幂等号，不覆盖旧交易。旧尝试、重试、回调与退款均可独立审计；支付创建结果不确定时只用同一个尝试号查单或补偿，不会另起一笔扣款。创建、关闭和成功回调共用订单级互斥锁；旧尝试异常迟到成功时会先关闭仍待支付的新尝试，并把订单置为 `PAYMENT_EXCEPTION` 产生错误告警；如果不同支付尝试仍先后成功，同样进入异常处置，不能按普通幂等回调静默吞掉。

会员卡等级、开卡记录和等级订单在 V1 是可管理的后台数据；“首单自动开卡”“消费成长值”“储值赠成长值”仅保留数据结构，不会在当前订单生命周期中自动执行，运营页面已明确标注并禁用这些规则。等真实支付回调与顾客微信身份闭环完成后再开放，避免出现界面显示已启用、实际却没有发放权益的情况。

## 3. 店铺装修

V2 采用受控组件装修器，不接受商户输入 HTML、CSS、JavaScript 或任意小程序路径。装修页覆盖：

- 页面装修；
- 全店风格；
- 点单风格；
- 导航设置；
- 启动页；
- 素材库；
- 版本记录。

首页组件支持门店头部、轮播图、公告、快捷入口、图片、文本和留白。点单风格可控制分类位置、商品列表/宫格、描述/库存/售罄展示与页面密度。V1 使用微信原生 TabBar：首页、点单、订单和“我的”四个入口及顺序固定，仅允许修改文案和导航色板；隐藏与拖拽排序保留为后续能力。素材库支持 JPG、PNG、GIF 上传、分组、缩略图选择和引用保护；门店 Logo、装修、广告、客服二维码、加料图片与会员卡面均通过缩略图选择器使用素材，不再向商户直接展示内部图片 URL。

### 3.1 草稿与发布

- 保存草稿使用乐观锁 `expectedRevision`，并发修改返回 `DRAFT_CONFLICT`；
- 顾客端永远只读取已发布的不可变版本，不会读到未完成草稿；
- 发布在事务中创建新版本并原子切换指针；
- 回滚不会覆盖历史，而是从旧快照创建一个新的发布版本；
- 尚未发布装修时，以门店 Logo、Banner 和公告生成兼容配置。

## 4. 主要接口

```text
商品配置
GET/POST/PUT/DELETE /merchant/catalog-resources
GET/POST/PUT/DELETE /merchant/modifier-items
GET/POST/PUT/DELETE /merchant/modifier-groups
GET/PUT             /merchant/products/{id}/configuration

用户会员储值
GET/POST/PUT/DELETE /merchant/customers
GET/POST/PUT/DELETE /merchant/customer-tags
GET                 /merchant/member-summary
GET/POST/PUT/DELETE /merchant/member-levels
GET/PUT             /merchant/membership-settings
GET/POST            /merchant/member-card-issuances
GET/POST            /merchant/member-level-orders
GET                 /merchant/balance-ledger
POST                /merchant/customers/{id}/balance-adjustments
GET/POST/PUT/DELETE /merchant/stored-value-rules
GET/PUT             /merchant/stored-value-settings
GET/POST            /merchant/stored-value-records

店铺装修
GET/PUT              /merchant/decoration[/draft]
POST                 /merchant/decoration/publish
GET                  /merchant/decoration/versions
POST                 /merchant/decoration/versions/{id}/rollback
GET/POST/PUT/DELETE  /merchant/media-assets
```

所有商户接口从 JWT 身份取得 `tenant_id`，关联资源同时校验租户和门店归属，不接受客户端指定另一个租户。

## 5. 后续进入生产前的必要工作

1. 用真实微信 `wx.login` 建立租户范围内的顾客会话，订单顾客身份只从服务端 Token 注入；
2. 接入支付机构真实储值/等级支付能力后，只由验签回调或主动查单落账；
3. 素材改为 OSS/COS 签名直传、域名白名单和内容安全审核；
4. 为套餐、会员价、优惠券增加统一 Quote/Price Snapshot；
5. 补充并发余额、跨租户、装修发布和商品选项的数据库集成测试。
