# 功能可用性与发布前清理清单

本文约束商户后台和顾客小程序中“尚未开通、仅供测试、仅保存资料”的能力如何展示。目标是让商户和顾客只看到可理解、可执行的信息，不看到 `Mock`、`V1`、接口、数据库、扫码参数、多租户路由等实现细节，也不把尚未形成业务闭环的页面描述成已经可用。

## 1. 统一管理位置

- 商户后台注册表：`apps/merchant-web/src/features/availability/copy.ts`
- 商户后台提示组件：`apps/merchant-web/src/components/FeatureAvailabilityNotice.tsx`
- 仅开发环境可见提示：`apps/merchant-web/src/components/DeveloperOnlyNote.tsx`
- 顾客小程序注册表、弹窗和安全错误映射：`apps/customer-miniapp/miniprogram/utils/availability.ts`

新增未开放能力时，先在注册表登记，再由页面引用。不要在各页面重复编写“预留、后续接入、V1、Mock”等文字。技术诊断只能放进 `DeveloperOnlyNote`、日志或工程文档；正式构建不能展示。

## 2. 展示规则

| 状态 | 商户后台 | 顾客小程序 |
| --- | --- | --- |
| 已开放且形成闭环 | 展示配置和真实状态 | 展示入口与真实资产 |
| 商户可维护、顾客不可使用 | 明确说明当前能做什么、不能做什么 | 隐藏入口，或显示稳定的“暂未开通”说明 |
| 完全未实现 | 卡片可显示“敬请期待”，不可提供可保存的假开关 | 默认隐藏；产品明确要求占位时才显示友好说明 |
| 开发测试能力 | 仅开发构建展示 | 不向顾客展示 |
| 请求或系统异常 | 展示可操作的重试信息 | 使用安全错误映射，禁止透传后端、数据库或微信 SDK 原文 |

“暂未开通”不是错误，也不应承诺具体日期。文案优先说明顾客当前可选择的替代方式，例如“可选择堂食或到店自取”。

## 3. 当前已集中登记的边界

商户后台登记了外卖、正式小程序码、在线储值、余额付款、优惠券抵扣、抽奖权益、会员自动成长、付费会员、套餐销售、配置模板、加料库存、自动补库存、云打印、餐后结账、自动接单、语音提醒、顾客催单、自动退款、自提核销、顾客评价和服务号通知。

顾客小程序登记了外卖、储值、余额付款、优惠券抵扣、抽奖权益、会员服务和通用未开通服务。网络失败、服务异常、二维码异常等公共文案也由同一文件维护；原始服务错误仅用于诊断，不能直接进入 `Toast`、`Modal` 或页面错误区。

## 4. 正式营业前必须关闭或补齐

以下项目属于发布检查项。没有形成真实闭环时，必须保持入口关闭或测试能力不可见：

1. **支付测试模式**：`config/env.ts` 仍为 `paymentMode: "mock"`，结算页还会调用 `mock-confirm`。真实营业前必须切换到已验签、已查单的支付通道，彻底禁止顾客订单自动模拟支付成功。
2. **无扫码参数的默认门店**：当前演示小程序默认进入已完成装修和商品配置的 `manong-coffee-gulou`。多商户发布前必须确定无门店参数启动时进入平台选店页，还是提示顾客重新扫码；不得让其他商户顾客误入码农咖啡。
3. **正式小程序码**：当前正式构建只显示“点单码正在准备中”。必须接通微信小程序码生成、下载和失效管理后，才能交付桌牌、快餐码牌和门店宣传物料。
4. **优惠券资产与核销**：当前顾客券包含本地领取记录，尚不能作为可信订单抵扣凭证。开放前必须完成顾客身份、服务端券资产、结算计价、核销、退款返券和过期处理；未完成时禁止宣传“可抵扣”。
5. **抽奖权益**：活动结果可记录，但优惠券奖品的使用依赖上一项闭环。未完成时只能展示活动说明，不能承诺奖品可直接抵扣订单。
6. **在线储值和余额付款**：当前只能登记已线下收款的储值记录。开放前必须完成支付、到账、余额账本、并发扣减、退款、对账和顾客身份；不得把线下登记页当成在线充值。
7. **会员自动化与付费会员**：首单开卡、消费成长、自动升级和付费开卡尚未执行。开放前必须接入可信订单与会员身份，清理固定昵称、余额、积分和等级等假资产展示。
8. **云打印设备**：模拟打印机只允许开发环境创建。正式营业前需接通设备厂商、确认 SN 绑定、签名、状态查询、失败重试和补打；生产数据中不得残留模拟设备。
9. **微信服务号通知**：必须完成平台服务号参数、消息模板、商户接收人授权绑定、发送结果和失败重试，才能标记为已启用。
10. **隐私政策与用户协议**：每个营业商户必须配置与实际收集信息一致的文本；空占位文本不能通过发布检查。
11. **商品扩展能力**：套餐销售、模板自动套用、加料独立库存和每日自动补库存未开通。相应开关保持禁用，商户维护的资料不能暗示会自动作用于价格或库存。
12. **未开放营销、排队和预约入口**：商户端可显示“敬请期待”；顾客端点击必须返回统一未开通提示，不能进入空白页或只筛出空订单。
13. **外卖**：允许产品在不含外卖的范围内正式发布，但所有外卖开关必须禁用，订单创建必须拒绝外卖类型，顾客端统一提示可选择堂食或到店自取。

## 5. 发布验收

- 使用生产构建访问商户后台，确认所有 `DeveloperOnlyNote` 均不可见。
- 全局检索顾客可见模板和页面，确认没有 `Mock`、`V1`、联调、数据库、接口、`scene` 参数或“多商户共用”等词。
- 人为制造断网、HTTP 500、后端英文错误和微信合法域名错误，确认顾客只看到统一友好提示，原始错误只进入诊断日志。
- 逐项核对本文件第 4 节；对未完成项目验证入口隐藏或禁用，并验证服务端不能绕过页面创建相关交易。
- 执行：

  ```bash
  npm run test --workspace @tanban/merchant-web
  npm run test --workspace @tanban/customer-miniapp
  npm run build --workspace @tanban/merchant-web
  npm run typecheck --workspace @tanban/customer-miniapp
  ```
